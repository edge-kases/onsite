⏺ High-Level Architecture: Hub-Agent Model

  The core tension is central control vs. customer environment diversity
  (especially air-gapped). This naturally leads to:

  ┌─────────────────────────────┐
  │     Control Plane (Phylo)   │
  │  ┌───────┐ ┌─────────────┐  │
  │  │  API  │ │  Deployment  │  │

  │  └───┬───┘ └──────┬──────┘  │
  │      │    ┌───────┴───────┐  │
  │      └────┤  State Store  │  │
  │           └───────────────┘  │
  └─────────────┬───────────────┘
                │ (outbound from agent, not inbound)
      ┌─────────┼─────────┐
      ▼         ▼         ▼
  ┌────────┐┌────────┐┌────────┐
  │ Agent  ││ Agent  ││ Agent  │
  │(Cust A)││(Cust B)││(Cust C)│
  │ AWS    ││ GCP    ││ On-prem│
  └────────┘└────────┘└────────┘

  Two key architectural bets:

  1. Pull-based, not push-based. Agents inside customer environments poll the
  control plane for desired state. This solves air-gapped/firewall constraints —
   customer never opens inbound ports. Control plane never needs to reach into
  customer networks.
  2. Declarative desired-state reconciliation, not imperative commands. Control
  plane says "Customer A should be on v2.3.1." Agent continuously reconciles
  toward that. This makes the system self-healing — if agent disconnects and
  reconnects, it just catches up. No lost commands.

  What runs where:
  - Control plane (Phylo): tenant registry, version/deployment state, rollout
  orchestration, API
  - Customer env: lightweight agent that pulls desired state, executes
  deployments locally, reports back status/health

  - What happens during prolonged disconnection? (agent keeps running current

  - How do we get container images to air-gapped envs? (separate concern — image

⏺ Good bones, but I want to challenge a few things:

  1. The Bootstrap Problem — biggest risk

  You said "agent upgrades itself." This is changing the engine while the car is
   running. If the new agent version is buggy and crashes on startup, the agent
  is dead — and a dead agent can't roll itself back or talk to phylod for the
  last stable version.

  You need a supervisor/watchdog pattern:
  Customer K8s
  ┌──────────────────────────┐
  │  phylo-supervisor (thin,  │  ← rarely updated, ~never changes
  │  dumb, long-lived)        │
  │    │                      │
  │    ├── manages lifecycle  │
  │    ▼                      │
  │  phylo-agent (v2.3.1)    │  ← the thing that actually upgrades
  │    │                      │
  │    ▼                      │
  │  phylo-platform (v5.1)   │  ← the product being deployed
  └──────────────────────────┘

  Supervisor starts agent, watches health. If agent fails post-upgrade →
  supervisor rolls back to previous agent binary. This is how kubelet (managed
  by systemd), Datadog agent, etc. all work. Without this, a bad agent release
  bricks every customer.

  2. Agent upgrade ≠ Platform upgrade

  Are we conflating two things?

  - Agent upgrade: new version of the control/deployment software
  - Platform upgrade: new version of Phylo's actual product

  These should be separate deployment tracks with independent versioning. Agent
  upgrades are higher risk (bootstrap problem above). Platform upgrades are the
  bread-and-butter operation. Mixing them means an agent bug blocks platform
  rollouts.

  3. "Loop through all agents" — eager vs lazy

  The proposed flow: new version → phylod loops all agents → writes desired
  state per agent. At 1000 tenants this is a write storm on every release.

  Alternative — lazy computation: Don't precompute desired state. When agent
  polls, phylod resolves desired state on-the-fly from (agent config, latest 
  version, rollout policy). Benefits:
  - No fan-out write storm
  - Rollout policies (canary 5% → 25% → 100%) become trivial — just logic at
  read time
  - Scales better

  Tradeoff: less explicit audit trail of "what was intended when." But you can
  log the computed result when served.

  4. Rollback trigger — who detects failure?

  "If agent failed to upgrade, it talks to phylod" — how does it know it failed?
   What's the health check? And per point #1, if the agent itself is the broken
  thing, it can't self-report.

  You need two failure detection paths:
  - Agent self-reports (fast path): agent runs post-deploy health checks,
  detects failure, requests rollback
  - phylod detects silence (slow path): if agent stops polling within expected
  window, phylod marks it degraded. Can't fix it remotely (air-gap), but at
  least triggers an alert for Phylo ops

  What I'd keep as-is

  - Pull-based model — correct for the constraints
  - Auto-upgrade vs manual-upgrade toggle — essential for enterprise compliance
  - Desired-state reconciliation — solid pattern
  - Rollback to last-known-stable — right instinct

  What's your take on the supervisor layer and the agent-vs-platform split?
  Those feel like the two sharpest architectural decisions to nail down before
  building.

  The insight: K8s IS your supervisor

  Since agents run in K8s, don't add a supervisor — use K8s itself:

  Agent sees new desired version
          │
          ▼
  Agent updates its own K8s Deployment image tag
          │
          ▼
  K8s RollingUpdate kicks in
  (maxSurge:1, maxUnavailable:0)
          │
     ┌────┴────┐
     ▼         ▼
  New pod    Old pod STAYS ALIVE
  starts     until new pod passes
             readiness probe
     │
     ├─ Probe passes → K8s kills old pod → success
     │
     └─ Probe fails  → old pod stays → old agent reports failure to phylod

  No new component. K8s is already solving the "don't kill old until new is
  proven" problem.

  What phylod can do for healing

  phylod can't push, but it can prepare the rollback intent so the next poll
  picks it up:

  1. Detect — agent stops heartbeating → phylod marks it degraded after timeout
  2. Decide — phylod sets desired state back to last stable version
  3. Wait — even a crash-looping agent restarts periodically (K8s
  CrashLoopBackOff). On restart, agent polls phylod, gets rollback instruction,
  self-heals
  4. Alert — if agent stays silent beyond threshold, page Phylo ops

  So phylod's healing role: detect → set rollback intent → alert. Execution
  still happens in customer env via K8s, but phylod made the decision.

  The truly bricked case

  New agent can't start at all — never polls, never heals. This is unrecoverable
   without manual intervention regardless of architecture. Even a supervisor
  pattern fails if the underlying K8s node is down.

  Mitigation: canary rollouts. Don't upgrade all agents at once. Upgrade 1 →
  verify → 5% → verify → 100%. A bricked agent in canary stops the rollout
  before it hits the fleet.

  So: no arch change needed

  - K8s handles the supervisor role (rolling update + probes)
  - phylod handles detection + rollback intent
  - Canary rollouts limit blast radius
  - Manual intervention is the last resort (with runbooks + alerts)

  The one thing to nail: readiness probe must be meaningful — not just "process
  is running" but "agent can reach phylod and perform core ops." That's the
  difference between catching real failures vs letting a half-broken agent
  through.

  Does this address your concern, or do you want to handle a specific failure
  scenario differently?
